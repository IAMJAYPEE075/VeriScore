Inside VeriScore

Why the tool exists, what it measures, and how OpenGradient makes the result verifiable.

Crypto wallets are pseudonymous by design, but their on-chain behavior still leaves a pattern. VeriScore was built to turn that pattern into a trust signal and pair it with OpenGradient TEE inference so the explanation can be checked, not just believed.

Why It Exists

VeriScore was built to give wallets a readable trust layer.

Most wallet addresses are just strings. You can see transfers on a block explorer, but that does not immediately tell you whether a wallet is experienced, active, risky, dormant, deeply engaged in DeFi, or showing signs of stress. VeriScore exists to turn raw activity into a signal people can act on.

  • For non-technical users, it turns noisy wallet history into a simple score and explanation.
  • For technical users, it exposes the underlying trust factors such as activity breadth, repayment behavior, and liquidation risk.
  • For integrators, it adds a verifiable attestation layer so the result is not just another opaque API claim.
Signals

What the score is actually measuring

The score is based on observable wallet behavior, not social reputation or off-chain identity.

Wallet ageOlder wallets tend to provide stronger historical context than brand-new ones.
Transaction depthTotal on-chain activity helps separate throwaway wallets from established users.
Protocol breadthUsing multiple serious protocols suggests broader ecosystem participation.
Repayment historyRepayment behavior acts as a positive trust signal in lending-style contexts.
LiquidationsLiquidation activity is treated as a risk penalty rather than a neutral event.
Recency and flowRecent activity and stablecoin volume help distinguish dormant wallets from active ones.
Data Layer

Where the wallet intelligence comes from

VeriScore uses chain-specific indexers, then normalizes the output into one common scoring model.

SolanaHelius enhanced transactions are used to read wallet history, protocol interactions, activity timing, and transaction context.
EVMEtherscan V2 provides broad account and token activity, while Alchemy decoded APIs help identify real protocol usage, repayment flows, and liquidation-like events.
NormalizationAll chain data is translated into a shared metric shape so one scoring system can work across networks.
How It Works

A wallet score moves through four stages before it reaches the interface.

The product is designed so the final number is not magic. It follows a traceable path from wallet input to verifiable output.

Wallet pasted -> chain is selected or auto-detected -> on-chain activity is fetched and normalized -> deterministic score is computed from behavioral signals -> OpenGradient TEE inference generates the final explanation -> attestation metadata is returned with the score
  • The first half is data engineering: fetching wallet history, decoding behavior, and reducing it to stable metrics.
  • The second half is trust packaging: generating a readable explanation and attaching proof-style metadata when OpenGradient returns it.
  • The result is meant to feel simple on the surface while still being defensible underneath.
OpenGradient

What OpenGradient is doing inside the tool

OpenGradient is not just being used as a generic model endpoint here. It is the trust layer that turns a wallet explanation into something attestable.

  • The backend first prepares a normalized wallet analysis payload from raw chain activity.
  • That payload is sent into OpenGradient for inference inside a Trusted Execution Environment.
  • The tool captures returned metadata such as payment hash, proof, tee attestation, or receipt references.
  • When a proof-like field exists, VeriScore builds a proof URL that can be checked on the OpenGradient explorer.
Why TEE Matters

Why this is different from a normal scoring API

A normal API can return any explanation it wants, and the caller has to trust the operator. VeriScore is built around the idea that trust should be inspectable.

  • The score is not only produced, it is accompanied by metadata that shows where and how the inference happened.
  • This makes the explanation more credible for lenders, partners, dashboards, and external reviewers.
  • It also gives the frontend a way to show proof status instead of pretending every result is equally trustworthy.
User Experience

What happens when someone pastes a wallet

The homepage is designed to keep the experience simple even though the backend work is multi-step.

Error handlingThe UI now shows real errors instead of inventing fallback demo scores.
Limits

What the tool can and cannot claim today

VeriScore improves wallet readability, but it is not trying to present itself as a perfect oracle.

  • EVM chain detection is best effort because a raw 0x address does not reveal its original network.
  • The decoded protocol registry is curated, which makes it more useful than heuristics but not yet exhaustive.
  • If OpenGradient is unavailable or underfunded, deterministic scoring still works, but proof fields may be empty.
  • The purpose of the tool is informed trust estimation, not a promise of identity, solvency, or future behavior.
